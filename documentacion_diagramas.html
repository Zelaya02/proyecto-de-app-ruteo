3.2 Fichas Técnicas Normalizadas (Copiar a Word)

Tabla 1: Caso de Uso – Iniciar Sesión
Caso de Uso CU-01 Iniciar Sesión
Actor Administrador / Despachador
Descripción Permite a un operador logístico autenticarse de forma segura en la plataforma para acceder al panel administrativo y sus herramientas de distribución.
Pre-condiciones • El usuario debe estar previamente registrado en el sistema.
• Disponer de conexión a internet o red local activa.
• El servidor backend de NEXO debe estar en ejecución.
Flujo de pasos 1. El operador ingresa a la pantalla de login (login.html).
2. El sistema muestra el formulario con los campos de usuario y contraseña.
3. El operador ingresa sus credenciales y presiona "Ingresar".
4. El sistema valida las credenciales contrastándolas en la base de datos PostgreSQL utilizando la librería BCrypt.
5. El sistema genera el token de sesión.
6. El sistema redirige al operador al Dashboard del panel principal (index.html).
Post-condiciones • La sesión del operador queda iniciada.
• Se habilita el temporizador de cierre automático por inactividad tras 5 minutos.
Excepciones • Usuario o contraseña incorrectos: El sistema deniega el acceso y lanza una alerta visual.
• Servidor apagado: El navegador muestra error de conexión de red.
Tabla 2: Caso de Uso – Gestionar Recursos de Flota y Clientes
Caso de Uso CU-02 Gestionar Recursos de Flota y Clientes
Actor Administrador / Despachador
Descripción Permite registrar, actualizar, deshabilitar lógicamente y listar clientes (con geolocalización), vehículos y choferes disponibles.
Pre-condiciones • El operador debe estar debidamente autenticado en el sistema (CU-01).
• Los datos geográficos de los clientes deben corresponder a coordenadas decimales válidas.
Flujo de pasos 1. El operador ingresa a las pestañas correspondientes en el panel (index.html).
2. El sistema muestra el listado de recursos actuales con la opción de añadir nuevo.
3. El operador presiona "Agregar" y digita la información solicitada en el formulario modal.
4. El sistema valida que los campos clave (como el nombre del cliente o chapa del vehículo) no existan previamente.
5. El sistema almacena los datos en la tabla correspondiente de PostgreSQL.
6. El sistema refresca dinámicamente el listado en la UI sin necesidad de recargar la página.
Post-condiciones • El recurso queda registrado en la base de datos y disponible para los algoritmos de optimización de rutas.
Excepciones • Duplicidad de claves: El sistema cancela el guardado si el nombre de cliente o matrícula ya existen.
• Campos incompletos: Se solicita al operador rellenar los inputs obligatorios.
Tabla 3: Caso de Uso – Configurar Reglas de Ruteo
Caso de Uso CU-03 Configurar Reglas de Ruteo
Actor Administrador / Despachador
Descripción Permite establecer y flexibilizar los límites de capacidad de paradas por móvil según la categoría de venta del cliente.
Pre-condiciones • El operador cuenta con sesión activa en el sistema.
Flujo de pasos 1. El operador navega hacia la sección "Reglas de Ruteo" (reglas.html).
2. El sistema despliega las reglas activas con sus categorías y topes máximos por móvil.
3. El operador modifica los valores numéricos o activa/desactiva la regla mediante el toggle de flexibilidad.
4. El operador presiona "Guardar Cambios".
5. El sistema valida los datos de entrada y sobrescribe las reglas en la tabla reglas_ruteo.
Post-condiciones • Los límites y estados modificados se aplican dinámicamente al generar la siguiente ruta.
Excepciones • Valor inválido: El sistema no permite guardar si se ingresa un límite negativo o vacío.
Tabla 4: Caso de Uso – Generar y Optimizar Rutas
Caso de Uso CU-04 Generar y Optimizar Rutas
Actor Administrador / Despachador, OpenRouteService API (Ext)
Descripción Agrupa de forma automatizada los clientes seleccionados en camiones asignados, ordenándolos óptimamente por su cercanía de carreteras.
Pre-condiciones • Deben existir clientes activos geolocalizados y choferes con vehículos libres.
• Conexión a internet estable y API Key de OpenRouteService configurada.
Flujo de pasos 1. El operador ingresa a "Planificación de Ruta" (ruta.html).
2. El operador marca en la lista interactiva los clientes a visitar y los choferes/vehículos habilitados.
3. El operador presiona "Generar Ruta Optimizada".
4. El backend inicia el cálculo heurístico y realiza consultas a OpenRouteService API para obtener matrices de distancias en carretera.
5. OpenRouteService retorna las distancias reales (m) y tiempos estimados de conducción (s).
6. El sistema estructura la ruta y las entregas ordenadas en la base de datos con un token único de seguimiento.
7. El sistema renderiza los recorridos en el mapa interactivo y muestra un resumen estadístico.
Post-condiciones • Se crean las rutas y entregas asociadas en estado "Pendiente" y listas para despacho.
Excepciones • Falla de API de mapas: Si OpenRouteService falla, el sistema realiza un fallback a distancias lineales por Haversine de manera transparente, notificando al operador.
Tabla 5: Caso de Uso – Despachar Ruta por WhatsApp
Caso de Uso CU-05 Despachar Ruta por WhatsApp
Actor Administrador / Despachador, WhatsApp Web / API (Ext)
Descripción Envía un mensaje formateado con las direcciones de entrega de la ruta y los links de localización GPS directamente al teléfono del chofer.
Pre-condiciones • La ruta del móvil correspondiente debe estar generada y persistida.
• El conductor asignado debe tener guardado un número de teléfono válido.
Flujo de pasos 1. El operador visualiza el itinerario generado en la sección de "Resultados de Ruta" (resultados.html).
2. El sistema despliega un botón con el icono de WhatsApp al lado del móvil asignado.
3. El operador presiona el botón "Enviar por WhatsApp".
4. El sistema compila la información (nombre del chofer, vehículo, secuencia ordenada de clientes y sus URLs directas a Google Maps) y crea un mensaje estructurado.
5. El sistema redirige al operador a la API Web de WhatsApp con el texto codificado en la URL.
6. El operador confirma el envío en la aplicación externa de WhatsApp.
Post-condiciones • El chofer asignado recibe en su terminal móvil la hoja de ruta con acceso directo para navegación GPS.
Excepciones • Falta número telefónico: El sistema muestra una alerta de error si el chofer no cuenta con un teléfono registrado.
Tabla 6: Caso de Uso – Registrar y Monitorear Entrega
Caso de Uso CU-06 Registrar y Monitorear Entrega
Actor Chofer / Repartidor, Administrador / Despachador
Descripción Permite monitorear el progreso de las rutas y registrar el resultado de cada visita individual de entrega por parte de los choferes en calle.
Pre-condiciones • La ruta de reparto debe estar activa (no finalizada).
• Acceso a internet en el dispositivo móvil del chofer.
Flujo de pasos 1. El chofer accede al módulo de "Seguimiento" (seguimiento.html) ingresando el token de su ruta.
2. El sistema despliega las entregas ordenadas por visitar.
3. Al visitar un punto, el chofer hace clic en "Actualizar Estado" de la entrega correspondiente.
4. El chofer selecciona el estado final (Entregado, Rechazado) y, si es rechazado, añade una observación explicativa.
5. El sistema valida el cambio de estado, actualiza la fecha en PostgreSQL y refresca dinámicamente el progreso en el panel del despachador.
6. Al completar todas las entregas, la ruta se marca como "Finalizada" (bloqueándose para cambios).
Post-condiciones • El estado de las entregas y las rutas se sincronizan en tiempo real para el control logístico.
Excepciones • Ruta Finalizada: El sistema impide modificar estados si la ruta fue marcada previamente como cerrada/finalizada.
• Pérdida de red: El dispositivo guarda el estado una vez que recupera señal.
